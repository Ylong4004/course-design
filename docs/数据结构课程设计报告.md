# 1 题目简介与设计目的

撰写人：架构设计负责人 光雨晨

## 1.1 题目简介

本系统面向区域城市交通网络开展建模与综合分析。将参与路网的每一座城市抽象为图的**顶点**，城市之间连通的道路抽象为**带权边**，道路长度、通行耗时等作为边的**权值**，支持单向/双向通行，灵活构建有向带权图或无向带权图。

系统同时使用**邻接矩阵**和**邻接表**两种图存储结构，集成**深度优先遍历（DFS）**、**广度优先遍历（BFS）**、**Dijkstra 最短路径**、**Floyd 多源最短路径**、**Prim 最小生成树**、**Kruskal 最小生成树**、**拓扑排序**七大经典图算法，并手写实现了**循环队列**、**顺序栈**、**最小堆优先队列**、**并查集**四种基础数据结构。系统提供数字菜单和命令行两种交互模式，支持路网编辑、最短路径规划、最小生成树构建、拥堵仿真模拟、存储结构性能对比、数据持久化等全维度分析功能。

## 1.2 设计目的

1. **知识巩固**：综合运用线性表、栈、队列、树、图、排序、查找等数据结构核心知识，结合城市交通路网实际业务场景进行实践应用。
2. **算法能力训练**：手写实现图论经典算法，掌握 DFS、BFS、Dijkstra、Floyd、Prim、Kruskal、拓扑排序的算法原理与代码实现，具备算法复杂度分析能力。
3. **工程实践能力**：采用模块化设计思想，完成系统设计→编码→测试→文档全流程，体验多人协作开发模式。
4. **数据结构对比认知**：通过邻接矩阵与邻接表的双实现并行对比，直观理解不同存储结构在内存占用、遍历速度、查找效率方面的差异与适用场景。

# 2 问题描述与需求分析

撰写人：架构设计负责人 光雨晨

## 2.1 问题详细描述

本系统面向区域城市交通网络建模，需要对城市和道路进行抽象建模并开展多维度的网络分析。具体而言：

- **路网建模**：城市顶点支持新增和删除，道路边支持新增、删除和权值修改。所有修改需在邻接矩阵和邻接表两套存储结构上**同步执行**，保证数据一致性。
- **图遍历**：使用 DFS 和 BFS 两种方式遍历整个交通网络，输出城市访问序列。需正确处理非连通图场景。
- **最短路径规划**：Dijkstra 算法计算单源最短路径（指定起点到其余所有城市的最优路线与总路程）；Floyd 算法计算多源最短路径（输出任意两城间的最短距离对照表）。
- **最小生成树**：Prim 和 Kruskal 两种算法分别构建路网最小生成树，计算连通所有城市的最低道路建设总造价。
- **拓扑排序与环路检测**：对有向路网执行拓扑排序，输出合法通行序列，检测环路以判断是否会引发交通死锁。
- **拥堵仿真**：修改指定道路权值以模拟拥堵/限速，备份原始权值，支持一键恢复，并对比拥堵前后最优路线变化。
- **性能对比**：统计两种存储结构在内存占用、遍历耗时、查找效率等方面的差异，输出横向对比表格。
- **数据持久化**：路网数据以 TXT 文件保存于 `data/` 目录，程序启动自动加载，支持多路网文件切换，退出时自动保存到当前文件。

- **健壮性**：全面拦截非法输入（重复编号、不存在的城市、自环边、负权值等），程序不得崩溃、闪退、死循环。全项目使用 `safe_new`/`safe_delete` 系列宏管理内存，杜绝悬空指针与内存泄漏。
- **易用性**：多级数字菜单导航 + 命令行交互双模式。操作提示清晰，格式化输出整齐美观，错误信息具体明确（区分"城市编号已存在"、"道路不存在"、"容量已满"等具体原因）。

## 5.2 核心算法设计与伪代码

### 5.2.1 Dijkstra 最短路径

```
算法 Dijkstra(G, start):
  for 每个顶点 v: dist[v] ← ∞, prev[v] ← -1, determined[v] ← false
  dist[start] ← 0
  for count = 1 to V:
      u ← 未确定顶点中 dist 最小的顶点
      if u == -1: break
      determined[u] ← true
      for u 的每个邻接顶点 v:
          if !determined[v] and dist[u] + weight(u,v) < dist[v]:
              dist[v] ← dist[u] + weight(u,v)   // 松弛
              prev[v] ← u
  return dist, prev
```

### 5.2.2 Floyd 多源最短路径

```
算法 Floyd(G):
  for i=0..V-1, j=0..V-1:
      dist[i][j] ← (边存在 ? weight(i,j) : ∞); next[i][j] ← (边存在 ? j : -1)
      dist[i][i] ← 0
  for k=0..V-1, i=0..V-1, j=0..V-1:
      if dist[i][k] + dist[k][j] < dist[i][j]:
          dist[i][j] ← dist[i][k] + dist[k][j]; next[i][j] ← next[i][k]
```

### 5.2.3 Kruskal 最小生成树

```
算法 Kruskal(G):
  收集所有边 → 按权值冒泡升序排序
  初始化并查集: parent[i] = i
  for 每条边 e = (u, v, w) 按序:
      if find(u) ≠ find(v):
          union(u, v); 将 e 加入 MST
          if |MST| == V-1: break
  if |MST| != V-1: return 图不连通
```

### 5.2.4 拓扑排序（Kahn 算法）

```
算法 Kahn(G):
  计算入度 indegree[]; 构建 id→index 映射
  for v where indegree[v]==0: queue.push(v)
  while queue 非空:
      v ← queue.pop(); 序列追加 v
      for v 的邻接点 w: indegree[w]--; if indegree[w]==0: queue.push(w)
  if seq_len < V: return 存在环; else: return 序列
```

## 5.3 关键业务逻辑实现思路

### 双存储同步

所有路网编辑操作均在 `RoadNetwork` 类中通过 `matrix_graph` 和 `list_graph` 两个 `GraphBase*` 指针同步执行。例如 `add_city` 同时调用两边 `add_vertex`。道路增删改采用同理。

### 算法与存储解耦

所有算法函数接受 `const GraphBase*` 参数，通过 `get_vertex_count()`、`get_neighbors()` 等虚函数操作图数据，不感知底层是邻接矩阵还是邻接表。同一份算法代码可复用于两种存储结构。

### 拥堵模拟对比

`CongestionSimulator` 在修改权值前备份原始值。`run_comparison` 分别在拥堵前后的路网上运行 Dijkstra，缓存距离数组，输出对比报告。

### CLI 命令分发

`CommandParser::dispatch()` 采用 if-else 链分派 24 条命令，`tokenize()` 按空格分割输入，`resolve_path()` 自动补全路径。命令行和菜单通过 `cli_run_menu()` 中的循环相互切换。

# 8 算法复杂度分析

撰写人：算法 & 核心功能负责人 高彪

| 算法              | 时间复杂度       | 空间复杂度 | 说明                                                    |
| ----------------- | ---------------- | ---------- | ------------------------------------------------------- |
| DFS（递归）       | O(V+E)           | O(V)       | visited 数组 + 递归调用栈；非连通图外层循环遍历所有顶点 |
| BFS（队列）       | O(V+E)           | O(V)       | visited 数组 + 手写 Queue；非连通图同样处理             |
| Dijkstra          | O(V²)           | O(V)       | 数组实现，每次选最小遍历 V 个顶点；含 id→index 映射    |
| Floyd             | O(V³)           | O(V²)     | 三重循环 + dist/next 两个 V×V 矩阵                     |
| Prim              | O(V²)           | O(V)       | key+parent+inMST 三个数组                               |
| Kruskal           | O(E·logE)       | O(V)       | 冒泡排序 O(E²) 可优化为快排；并查集 O(α(V)) ≈ O(1)   |
| Kahn 拓扑排序     | O(V+E)           | O(V)       | 入度表（数组）+ 手写 Queue                              |
| 邻接矩阵边查询    | O(1)             | —         | `matrix[i][j]` 直接索引                               |
| 邻接表边查询      | O(degree)        | —         | 遍历目标顶点的边链表                                    |
| 并查集 Find/Union | O(α(V)) ≈ O(1) | O(V)       | 路径压缩 + 按秩合并；α 为反阿克曼函数                  |

# 9 系统评价与总结

撰写人：交互 & 测试 & 汇总负责人 刘幸源

## 9.1 程序优点

1. **架构清晰**：分层设计，算法层通过 `GraphBase` 抽象接口与存储层解耦，CLI 模块通过服务层接口与菜单模块解耦。同一份算法代码可运行于邻接矩阵和邻接表两种结构。
2. **功能完备**：实现 9 项基础功能 + 6 项拓展功能（路网详情查看、自动保存、命令行模式、调试模式、安全宏体系、23 个自动化测试）。
3. **双交互模式**：数字菜单 + 命令行，两种模式可相互切换，满足不同使用习惯。
4. **基础结构手写**：队列、栈、并查集、最小堆优先队列全部从零实现，STL 风格 API。
5. **代码规范**：统一 `snake_case` 命名、Doxygen 注释全覆盖、`safe_new`/`safe_delete` 宏统一管理内存。
6. **测试充分**：23 个自动化测试用例覆盖合法/非法/边界/交叉验证四类场景，全部通过。
7. **用户体验**：错误提示具体到错误码和中文描述；文件路径简化为只填文件名自动补全；退出自动保存。
8. **可维护性**：`DEBUG_MODE` 编译开关控制算法调试输出，答辩时一键开启，不影响正式版本性能。

## 9.2 程序存在不足

1. **图类型固定**：当前无向图/有向图由代码初始化时确定，未提供运行时切换功能。
2. **Dijkstra 仅为基本版**：未启用 PriorityQueue 优化版，稀疏图时 O(V²) 不如 O((V+E)·logV)。
3. **CLI `save` 默认路径跟踪**：CLI 自动加载后未通过 `set_current_file()` 记录路径，当前已修复。
4. **无图形可视化**：仅控制台模式，未提供图形化路网展示（Qt GUI 模块已开发但不在本报告范围内）。

## 9.3 优化设想与拓展方案

1. 增加图类型运行时切换功能（菜单中一键切换无向 ↔ 有向）
2. Dijkstra/Prim 加入 PriorityQueue 优化版
3. 为优先队列添加 `decrease_key` 性能优化（当前仅支持小规模测试）
4. 增加按城市名称模糊查询的功能
5. Kruskal 排序改用快排替代冒泡排序以提升大数据集性能

## 10.2 个人心得体会

### 架构设计负责人 光雨晨

作为本次课程设计的架构负责人，我主要负责了项目框架搭建、全局类型定义、图抽象基类设计、文件读写公共接口以及程序入口的编写。从最初的选题分析到最终的系统集成，整个过程中最大的挑战在于前期设计阶段——如何定义一套既能覆盖全部功能需求、又足够简洁易用的统一接口。`GraphBase` 中 26 个纯虚函数的取舍经过了反复推敲：既要保证算法层可以通过通用接口操作不同存储结构，又不能过于庞杂导致实现困难。

流程图绘制也让我对系统的全局有了更深刻的理解。通过 15 张 Mermaid 流程图的绘制，每个算法的执行路径、模块间的调用关系变得清晰可见，这为后续的三人协作提供了共同的参照基准。在项目管理层面，统筹小组进度、协调模块接口对接、解决跨模块数据交互冲突，这些经历让我认识到软件工程中"规范先行"的重要性——统一的命名规则、编码规范、注释格式，是让三个人写的代码无缝拼接的前提条件。

### 算法 & 核心功能负责人 高彪

本次课程设计中，我负责了所有图算法和基础数据结构的实现。从邻接矩阵的二维数组操作到邻接表的链表管理，从 Dijkstra 最短路径到 Floyd 多源最短路径，从 Prim 和 Kruskal 的最小生成树到 Kahn 拓扑排序，覆盖了图论的经典算法体系。

最大的收获在于理解了"抽象接口"在多人协作中的价值——`GraphBase` 定义了 26 个纯虚函数，所有算法都通过这个接口操作图数据，完全不关心底层是矩阵还是链表。这使得算法代码可以一次性写完，在两种存储结构上复用，大大减少了重复劳动。

手写基础数据结构（队列、栈、堆、并查集）也让我深入理解了这些结构的内部原理。例如最小堆的 `sift_up`/`sift_down` 操作，以及并查集的路径压缩和按秩合并优化，这些细节是直接调用 STL 难以体会到的。调试过程中遇到的 `capacity` 同名冲突、`top` 成员与 `top()` 方法冲突等问题也加深了对 C++ 编译器规则的认知。

### 交互 & 测试 & 汇总负责人 刘幸源

作为交互与测试负责人，我主要负责了菜单系统、命令行模块、输入校验、输出格式化、拥堵模拟器、性能对比器以及全系统联调测试工作。在 UI 层开发中，我设计实现了 10 项主菜单和对应的多级子菜单，将用户输入的合法性校验、业务规则的拦截提示和格式化输出统一封装为独立的 `Validator` 和 `Formatter` 类，保证了交互界面的一致性和可维护性。

命令行模式的开发是整个项目中让我收获最大的部分。从最初的框架构思到最终实现 24 条可执行命令，我深入了解了命令解析、参数拆分、模式分派和单条批处理等多种交互形态的设计方法。菜单和命令行相互切换的循环逻辑在调试中经过多次修改，最终实现了用户可在两种模式之间无缝切换的体验。

全系统联调阶段，我负责了 23 个测试用例的编写和 TestRunner 框架的搭建。测试覆盖了合法、非法、边界和交叉验证四类场景，所有测试用例在修复了文件路径适配、MinGW 兼容性等问题后全部通过。BUG 修复记录的完整整理让我认识到测试驱动开发在实际工程中的重要性——每一个 BUG 从发现到定位再到修复，都依赖于充分的测试用例覆盖和对错误现象的细致观察。

在报告汇总阶段，我将三人各自负责的章节进行了统一的排版与整合，使整份课程设计报告在格式、风格和逻辑上保持连贯一致。这份经历让我理解了技术文档撰写在软件工程全流程中的位置——好的文档不仅是对已完成工作的记录，更是对团队协作过程中经验与教训的沉淀。
