## 三、光雨晨：架构与数据结构设计

### 1. 自己负责什么

光雨晨主要说：

- 项目整体架构设计
- `common/types.h`、`defines.h` 中的全局类型、错误码、安全内存宏
- `GraphBase` 抽象接口设计
- 邻接矩阵和邻接表的统一接口思路
- `RoadNetwork` 双存储同步管理
- `FileManager` 文件读写基础接口
- CMake / 程序入口 / 文档架构

### 2. 建议发言稿

> 我主要负责项目的架构设计和底层公共接口。这个系统的核心是图，所以我们首先定义了一个抽象基类 `GraphBase`，里面规定了添加顶点、删除顶点、添加边、删除边、获取邻接点、获取全部城市 ID 等统一接口。这样做的好处是算法层不需要关心底层到底是邻接矩阵还是邻接表，只要通过 `GraphBase` 调用接口即可。

> 在存储结构上，我们同时维护了邻接矩阵和邻接表两套数据。邻接矩阵适合边查询，查询两个城市之间是否有道路可以接近 O(1)；邻接表适合稀疏图，遍历邻接点更节省空间。为了保证两套结构数据一致，我们封装了 `RoadNetwork` 类，所有添加城市、删除城市、添加道路、修改道路权值的操作，都在矩阵和邻接表中同步执行。

> 公共类型方面，我们在 `types.h` 中定义了 `City_t`、`Edge_t`、错误码、图类型、存储类型等结构；在 `defines.h` 中定义了 `safe_new`、`safe_delete` 等安全内存宏，减少空指针和内存泄漏风险。

> 后期我们还对文件管理和图类型做了优化。原来路网类型在程序启动时固定，导致有向图 JSON 文件加载失败。现在加载文件时会先读取 JSON 中的 `graph_type` 字段，再通过 `RoadNetwork::reset()` 自动创建有向图或无向图，保证不同类型的路网都能正常加载。

### 4. 老师可能问的问题

**问：为什么要同时实现邻接矩阵和邻接表？**

答：

> 这是为了比较两种图存储结构的适用场景。邻接矩阵查询边很快，适合稠密图；邻接表更节省空间，适合稀疏图。城市交通路网通常比较稀疏，所以邻接表更节省内存，但矩阵在边查询和性能对比中也有价值。

**问：为什么要设计 `GraphBase`？**

答：

> 为了解耦算法层和存储层。DFS、Dijkstra 等算法只依赖 `GraphBase` 接口，不关心底层实现。这样同一套算法可以同时运行在邻接矩阵和邻接表上，代码复用性更好。

**问：有向图和无向图怎么处理？**

答：

> 图类型用 `GraphType` 枚举表示，JSON 文件中 `graph_type=0` 表示无向图，`graph_type=1` 表示有向图。添加边时，如果是无向图，底层会维护双向关系；如果是有向图，只保存 `from -> to` 方向。

## 四、高彪：核心算法与数据结构实现

### 1. 自己负责什么

高彪主要说：

- 邻接矩阵、邻接表具体实现
- 手写队列、栈、并查集、最小堆优先队列
- DFS / BFS
- Dijkstra / Floyd
- Prim / Kruskal
- 拓扑排序
- 城市 ID 到数组下标映射问题修复

### 2. 建议发言稿

> 我主要负责图存储结构和核心算法实现。图存储部分实现了邻接矩阵和邻接表两套结构。邻接矩阵内部通过二维数组保存边权，邻接表通过链表保存每个城市的出边。两种结构都实现了 `GraphBase` 中规定的统一接口。

> 算法部分，我们实现了 DFS 和 BFS，用于图遍历；Dijkstra 和 Floyd 用于最短路径；Prim 和 Kruskal 用于无向图的最小生成树；Kahn 算法用于有向图拓扑排序。为了满足课程设计要求，我们还手写了基础数据结构，包括循环队列、顺序栈、并查集和最小堆优先队列。

> 这里我重点讲一下 Dijkstra。它的思路是维护起点到每个城市的当前最短距离，每次从未确定的城市中选择距离最小的城市，再用它去更新邻接城市的距离。我们用 `dist` 数组保存最短距离，用 `prev` 数组保存前驱城市，最后可以回溯出完整路径。

> Kruskal 算法中，我们先收集所有边并按权值排序，然后用并查集判断加入某条边是否会形成环。如果不会形成环，就把这条边加入最小生成树，直到边数达到 `V-1`。

> 后期我们还修复了一个比较关键的问题：城市 ID 不一定连续，比如城市编号可能是 10、20、30，不能直接把 ID 当数组下标。因此我们把算法内部统一改成先通过 `get_all_vertex_ids()` 获取城市 ID 列表，再建立 ID 到数组下标的映射。这样非连续城市 ID 也能正确运行遍历、最短路径、最小生成树和拓扑排序。

### 3. 算法说明简短版

如果时间不够，可以这样压缩：

- DFS：递归或栈思想，沿着一条路走到底，再回溯
- BFS：使用队列，一层一层访问邻接城市
- Dijkstra：单源最短路径，每次确定当前最近城市
- Floyd：三重循环，枚举中转点，求所有城市对最短路径
- Prim：从一个点开始逐步扩展最小生成树
- Kruskal：按边权从小到大选边，用并查集避免成环
- 拓扑排序：统计入度，反复取入度为 0 的点

### 5. 老师可能问的问题

**问：Dijkstra 能处理负权边吗？**

答：

> 不能。Dijkstra 的前提是边权非负。本系统道路权值代表距离或通行时间，所以输入校验中要求权值必须是正整数。

**问：Prim 和 Kruskal 为什么不能用于有向图？**

答：

> 最小生成树是无向连通图上的概念，它要求用最小代价连接所有顶点。有向图对应的是最小树形图等其他问题，不属于 Prim/Kruskal 的适用范围，所以系统在有向图下会提示不支持。

**问：Floyd 和 Dijkstra 有什么区别？**

答：

> Dijkstra 是单源最短路径，给定一个起点，求它到其他所有点的最短距离；Floyd 是多源最短路径，可以求任意两个点之间的最短距离。Floyd 更通用，但时间复杂度是 O(V³)，适合顶点数量不太大的场景。

**问：为什么不能直接用城市 ID 做数组下标？**

答：

> 因为用户输入的城市 ID 不一定连续，也不一定从 0 或 1 开始。如果直接用 ID 做下标，可能越界或浪费空间。所以我们用城市 ID 列表建立逻辑下标映射。

**问：为什么从 TXT 改成 JSON？**

> `FileManager` 会解析并校验字段。如果缺少 `graph_type`、城市或道路字段，或者城市 ID 重复、道路端点不存在、权值非法，都会返回 `ERR_FILE_FORMAT`，上层菜单或 CLI 会提示加载失败。

## 八、老师常见问题与统一回答

### 1. 你们这个系统用了哪些数据结构？

答：

> 图采用邻接矩阵和邻接表双结构；BFS 和拓扑排序用队列；DFS 使用栈/递归思想；Kruskal 使用并查集；Dijkstra/Prim 可以使用最小堆优先队列思想。项目中这些基础结构都进行了手写实现。

### 2. 邻接矩阵和邻接表分别有什么优缺点？

答：

> 邻接矩阵边查询快，适合稠密图，但空间复杂度是 O(V²)。邻接表只保存存在的边，更适合稀疏图，遍历邻接点效率高，但查询任意两点是否相连需要遍历边链表。

### 3. 为什么道路权值不能为负？

答：

> 道路权值表示距离、耗时或成本，现实意义上应为正数。同时 Dijkstra 算法也要求边权非负，所以系统输入校验中限制权值必须为正整数。

### 4. 你们怎么保证邻接矩阵和邻接表数据一致？

答：

> 所有编辑操作都通过 `RoadNetwork` 统一入口执行。比如添加道路时，先添加到邻接矩阵，再添加到邻接表；如果第二步失败，会回滚第一步。删除和修改同理。

### 5. 有向图加载失败的问题怎么解决的？

答：

> 原来程序启动默认创建无向图，而 JSON 中如果 `graph_type=1` 是有向图，加载时类型不一致就失败。现在加载前先调用 `detect_graph_type()` 读取文件中的图类型，再用 `RoadNetwork::reset()` 重建为对应类型，最后再加载数据。

### 6. 如果城市 ID 是 100、200、300，算法还能跑吗？

答：

> 可以。我们没有直接把城市 ID 当数组下标，而是先获取全部城市 ID，建立 ID 到内部下标的映射。这样城市 ID 可以不连续。

### 7. JSON 文件中无向图和有向图有什么区别？

答：

> `graph_type=0` 是无向图，添加一条边时底层会按双向关系处理；`graph_type=1` 是有向图，只表示 `from` 到 `to` 的单向道路。

### 8. 你们项目最大的难点是什么？

可由不同人回答：

光雨晨：

> 最大难点是接口设计和模块解耦。需要让算法层、存储层、服务层、UI 层互相独立，但又能协作。

高彪：

> 最大难点是算法要适配任意城市 ID，而不能依赖连续数组下标。这个问题会影响多个算法。

刘幸源：

> 最大难点是菜单、CLI、文件加载和运行时状态同步。比如切换图类型后，模拟器和对比器也要重新绑定新的图对象。

### 9. 项目还有哪些不足？

答：

> Dijkstra 当前是数组版本，稀疏图下还可以进一步使用优先队列优化；JSON 解析器是针对本系统格式实现的，还可以增强错误提示；测试用例已有框架，但还可以进一步接入 CTest 做一键自动测试。
