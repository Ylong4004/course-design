## 二、完整个人发言稿

### 1. 开始部分

> 我是刘幸源，主要负责本项目的交互、文件管理和测试联调。我的工作目标是让底层图算法能够通过清晰的菜单和命令行被用户使用，同时保证输入合法、错误提示明确、文件保存加载可靠。

### 2. 菜单系统

> 菜单系统是普通用户使用项目的主要入口。我们设计了 10 项主菜单，包括路网编辑、图遍历、最短路径、最小生成树、拓扑排序、拥堵模拟、结构性能对比、数据文件管理、帮助和命令行模式。每个模块下面还有对应子菜单，用户通过输入数字完成操作。

> 菜单层本身不直接操作底层数组或链表，而是调用服务层和算法层接口。比如添加城市会调用 `RoadNetwork::add_city()`，执行 Dijkstra 会调用最短路径算法，保存文件会调用 `FileManager::save_to_file()`。

### 3. 输入校验和输出格式

> 为了避免非法输入导致程序崩溃，我把输入校验封装到了 `Validator` 中。比如读取菜单编号时会限制范围，读取城市 ID 和权值时会检查是否为合法整数，读取文件名时会检查长度。输出方面使用 `Formatter` 统一标题、分隔线、信息、警告、错误和成功提示，让界面风格保持一致。

### 4. CLI 命令行模式

> 除了菜单，我们还实现了 CLI 命令行模式。程序支持三种启动方式：无参数进入菜单模式，`--cli` 进入交互式命令行模式，或者直接跟命令参数进行单条批处理。比如 `network_console.exe dijkstra 1 5` 可以直接计算 1 到 5 的最短路径。

> CLI 中使用 `CommandParser::tokenize()` 拆分输入，再由 `dispatch()` 分发到不同命令处理函数。目前支持 25 条命令，包括 `new_city`、`new_road`、`dijkstra`、`topo`、`save`、`load`、`list`、`new_network` 等。

### 5. JSON 文件管理

> 文件管理是我这次修改比较多的部分。原来项目使用 TXT 格式保存路网，现在已经迁移为 JSON。JSON 文件包含 `graph_type`、`cities`、`roads` 三个核心字段。这样格式更清晰，也方便用户手动查看。

> 加载文件时，程序会先读取 `graph_type`。如果是 0，就按无向图加载；如果是 1，就按有向图加载。这样解决了以前把 `graph_type` 改为 1 后加载失败的问题。

> 另外，直接打开 `bin/network_console.exe` 时，程序工作目录是在 `bin`，以前找不到 `data` 文件夹。现在 `FileManager` 会自动尝试 `./data`、`../data`、`../../data`，所以从根目录、`bin` 或 `build/bin` 启动都可以正常找到 JSON 文件。

### 6. 新建路网功能

> 我们还新增了新建路网功能。菜单中可以进入“数据文件管理-新建路网”，输入文件名并选择 `0=无向图` 或 `1=有向图`，系统会生成一个空 JSON 文件。CLI 中也可以使用 `new_network 1 demo` 创建有向路网。新建成功后，当前内存路网也会切换到对应类型，后续可以继续添加城市和道路。

### 7. 拥堵模拟和性能对比

> 拥堵模拟本质上是临时修改某条道路的权值。系统会在修改前保存原始权值，恢复时再写回去。拥堵前后对比会分别运行最短路径算法，展示路径和距离变化。结构性能对比则会统计邻接矩阵和邻接表在内存占用、遍历耗时、顶点查找、边查询等方面的差异。

### 8. 测试和联调

> 测试方面，我们设计了 23 个测试用例，覆盖添加城市、添加道路、重复城市、自环边、负权值、遍历空图、最短路径、Floyd 与 Dijkstra 交叉验证、MST、拓扑排序、文件保存加载等情况。除了自动测试，我也做了手工联调，比如从 `bin` 目录运行程序、CLI 列出文件、加载有向图、新建路网等。

### 9. 小结

> 总结来说，我负责的部分主要是把底层算法和数据结构包装成用户可以稳定操作的系统功能，同时处理文件持久化、异常提示和测试验证。我的重点不是单个算法，而是保证整个系统可以完整跑起来、用户能用、出错能提示、数据能保存。

## 五、老师可能问的问题与回答

### Q1：菜单和 CLI 的区别是什么？

答：

> 菜单适合按功能逐步操作，CLI 适合快速命令和批处理。两者底层调用的是同一套服务和算法接口，所以功能一致。

### Q2：CLI 怎么解析命令？

答：

> 先用 `tokenize()` 按空格拆分输入，得到命令名和参数数组，然后 `dispatch()` 根据命令名调用对应处理函数。

### Q3：输入非法字符会怎么样？

答：

> 菜单模式下 `Validator` 会循环提示重新输入；CLI 模式下命令处理函数会检查参数个数和类型，不合法就输出用法提示。

### Q4：为什么要把 TXT 改成 JSON？

答：

> JSON 可读性更好，字段含义明确，不依赖固定行顺序。比如 `graph_type`、`cities`、`roads` 都很直观，也方便后续扩展。

### Q5：JSON 格式错误怎么处理？

答：

> `FileManager` 会检查必须字段、城市 ID、道路端点、权值、自环、重复边等。如果不合法，会返回 `ERR_FILE_FORMAT`，上层提示加载失败。

### Q6：新建路网时为什么先保存文件再重置当前路网？

答：

> 这样更安全。如果文件路径无效或没有写入权限，保存失败时不会清空当前正在编辑的路网。

### Q7：如果用户输入文件名不带 `.json` 怎么办？

答：

> CLI 和菜单都会自动补全 `.json` 后缀，并默认放到 `data/` 目录下。

### Q8：直接双击 `bin/network_console.exe` 为什么现在能找到数据？

答：

> 因为 `FileManager` 会检查多个候选路径，包括 `./data`、`../data`、`../../data`，从而适配不同启动目录。

### Q9：为什么要刷新拥堵模拟器和对比器指针？

答：

> 因为重置路网会释放旧图并创建新图。如果模拟器和对比器还保存旧图指针，就可能访问已经释放的对象，所以必须重新绑定。

### Q10：拥堵模拟为什么只改权值？

答：

> 在路网模型中，道路拥堵可以抽象为通行成本增加，因此修改边权是最直接的模拟方式。

### Q11：拥堵后怎么比较路径变化？

答：

> 分别在拥堵前和拥堵后运行 Dijkstra，比较距离和前驱路径。如果路径或距离发生变化，就说明拥堵影响了最优路线。

### Q12：性能对比怎么测？

答：

> 对邻接矩阵和邻接表分别统计内存占用、遍历耗时、顶点查找和边查询次数，然后输出对比结果。

### Q13：测试用例主要覆盖什么？

答：

> 覆盖正常添加、非法输入、边界情况、算法正确性和文件读写。例如重复城市、自环道路、负权值、空图遍历、Dijkstra、Floyd、MST、拓扑排序和文件保存加载。

### Q14：为什么 `ctest` 没有测试？

答：

> 当前项目测试框架是自定义 `TestRunner`，需要在 `main.cpp` 中开启 `run_all_tests()`。还没有把它注册成 CMake/CTest 标准测试目标，这是后续可以优化的地方。

### Q15：CLI 批处理有什么用？

答：

> 可以不用进入交互界面，直接执行一条命令，适合快速测试或演示。例如 `network_console.exe dijkstra 1 5`。

### Q16：如果老师问你负责的最大难点？

答：

> 最大难点是状态同步。比如加载不同图类型的文件时，不只是图数据要换，当前文件路径、模拟器、对比器、菜单和 CLI 的提示都要同步，否则会出现部分模块仍使用旧状态的问题。

### Q17：如果老师问你最满意的功能？

答：

> 我比较满意的是 JSON 文件管理和新建路网功能。因为它解决了实际使用中的问题：文件更清晰、可以直接创建有向/无向路网，也能从 `bin` 目录直接运行。

### Q18：如果老师问异常处理怎么做？

答：

> 底层返回统一错误码，上层菜单或 CLI 根据错误码输出具体提示。这样底层逻辑和用户界面不会混在一起。

### Q19：如果老师问为什么不用图形界面演示？

答：

> 项目中有 Qt GUI 模块，但控制台版本功能更完整、运行环境更稳定，所以课程汇报主要演示控制台和 CLI。Qt 可以作为拓展方向。

### Q20：如果老师问这个项目有什么不足？

答：

> JSON 解析器是针对本系统格式实现的，不是完整通用 JSON 库；测试框架还没有接入 CTest；部分算法比如 Dijkstra 还可以进一步用优先队列优化。

## 七、老师专项六问：交互、异常处理、测试与文档

### 1. 你设计了合法、非法、边界三类测试用例，请各举一例，说明预期输出与实际运行结果。

推荐回答：

> 合法用例可以举“添加城市”。输入一个不存在的城市 ID 和合法城市名，例如 ID=1、名称=City1，预期返回 `SUCCESS`，实际运行中城市成功加入，城市数量增加。  
>
> 非法用例可以举“重复添加城市”。先添加 ID=1 的城市，再次添加 ID=1，预期返回 `ERR_CITY_DUPLICATE`，实际运行中程序没有重复插入，并提示城市 ID 已存在。  
>
> 边界用例可以举“空图执行遍历”或“添加城市至容量满”。空图执行 DFS/BFS 时，预期返回 `ERR_GRAPH_EMPTY`，实际运行中程序不会崩溃，而是提示当前图为空。容量满时继续添加城市，预期返回 `ERR_GRAPH_FULL`，实际也会提示城市数量达到上限。

也可以准备这三个更贴近功能演示的例子：

| 类型 | 用例 | 预期结果 | 实际结果 |
| ---- | ---- | -------- | -------- |
| 合法 | `new_road 1 2 10`，两个城市存在 | 返回成功，道路数增加 | 成功添加道路 |
| 非法 | `new_road 1 1 10` | 返回 `ERR_SELF_LOOP` | 提示不允许自环边 |
| 边界 | `load 不存在文件` | 返回 `ERR_FILE_OPEN_FAIL` | 提示文件打开失败 |

### 2. 程序全部异常容错逻辑由你完善，列举 3 种典型非法输入场景，说明拦截判断代码实现思路。

推荐回答：

> 第一类是输入类型错误，比如菜单要求输入数字，但用户输入字母。`Validator::read_int_safe()` 会读取输入并判断是否是合法整数，不合法就提示重新输入。  
>
> 第二类是业务规则错误，比如添加道路时起点和终点相同，或者权值小于等于 0。这个在 `RoadNetwork::add_road()` 中会判断 `from == to` 返回 `ERR_SELF_LOOP`，判断权值非法返回 `ERR_INVALID_INPUT`。  
>
> 第三类是文件错误，比如用户加载不存在的文件或格式错误的 JSON。`FileManager` 会先尝试打开文件，失败返回 `ERR_FILE_OPEN_FAIL`；解析字段失败、城市重复、道路端点不存在、权值非法等情况返回 `ERR_FILE_FORMAT`。

可以再补一句：

> 我的处理思路是：底层只返回错误码，上层菜单和 CLI 根据错误码输出中文提示，这样逻辑和界面分离。

### 3. 联调时发现的最严重 BUG 是什么？由哪个模块产生，你如何配合对应组员完成复测修复？

推荐回答：

> 我认为比较严重的 BUG 是直接打开 `bin/network_console.exe` 时找不到 `data/` 文件。这个问题属于文件路径和运行目录的问题，主要发生在文件管理模块。因为用户双击 exe 时，当前工作目录是 `bin`，原来的代码只找 `./data`，所以加载失败。  
>
> 修复时我们把路径解析集中到 `FileManager`，让它依次尝试 `./data`、`../data`、`../../data`。我负责从用户使用场景复现问题，并在修复后从 `bin` 目录运行 `list`、`load`、`show_network` 进行复测。

也可以回答另一个严重 BUG：

> 另一个严重问题是 JSON 中 `graph_type=1` 的有向图加载失败。这个问题涉及文件模块、路网管理模块和交互层。最后的修复方案是加载前先 `detect_graph_type()`，再 `RoadNetwork::reset()`，然后重新绑定模拟器和对比器。我负责菜单和 CLI 入口的接入，以及用临时有向 JSON 文件复测。

### 4. 菜单多级交互、格式化输出是你优化的，设计时考虑了哪些用户操作体验问题？

推荐回答：

> 我主要考虑了五点。  
>
> 第一，菜单层级要清楚，主菜单按功能分区，子菜单只放当前模块相关操作。  
>
> 第二，所有子菜单都用 0 返回，用户不用记复杂命令。  
>
> 第三，错误提示要具体，比如区分城市不存在、道路重复、自环边、文件打开失败，而不是统一提示“操作失败”。  
>
> 第四，文件名输入要简化，用户只输入 `zhengzhou`，程序自动补成 `data/zhengzhou.json`。  
>
> 第五，输出格式要统一，用标题、分隔线、成功/错误/警告提示提升可读性，方便汇报演示。

### 5. 你整合全部代码生成完整程序，合并时出现过哪些代码冲突？如何统一编码规范、注释格式？

推荐回答：

> 合并时主要有几类冲突。第一是接口参数不一致，比如算法层需要 `GraphBase*`，服务层传矩阵还是邻接表需要统一；我们最后规定算法统一通过 `GraphBase` 接口调用。  
>
> 第二是路径和文件格式变化，原来部分代码和文档仍然使用 `.txt`，后面统一迁移到 `.json`，同时把 `list/save/load/auto_load` 的逻辑和文案都改掉。  
>
> 第三是编码和构建问题，Windows 下中文输出、PowerShell 写出的 UTF-8 BOM、CMake 是否依赖 Qt 等都需要统一处理。  
>
> 编码规范上，我们统一使用 `snake_case` 命名，函数返回错误码，注释尽量说明功能和参数，不在 UI 层直接访问底层结构。文档方面统一用 Markdown 标题、表格和代码块整理。

### 6. 报告中系统优缺点、优化设想由你撰写，结合运行测试结果，说说本系统最核心两处缺陷，给出可行改进方案。

推荐回答：

> 第一处缺陷是 Dijkstra 当前主要是数组版本，时间复杂度 O(V²)，在城市数量更大、道路较稀疏时效率不如堆优化版本。改进方案是使用我们已经手写的最小堆优先队列，把 Dijkstra 优化为 O((V+E)logV)。  
>
> 第二处缺陷是测试框架还没有完全接入 CMake/CTest。虽然项目中有 23 个 TestRunner 测试用例，但需要手动在 `main.cpp` 中开启测试入口。改进方案是把测试编译成单独目标，并注册到 CTest，这样可以用一条命令完成自动测试。  
>
> 如果再补一处，就是 JSON 解析器是针对本系统固定格式实现的，错误定位不够细。后续可以增强错误提示，或者在允许的情况下使用成熟 JSON 库。
