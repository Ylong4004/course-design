## 二、完整个人发言稿

### 1. 开始部分

> 我是高彪，主要负责本项目中的图存储结构和核心算法实现。这个项目本质上是对城市交通路网进行图建模，所以算法部分围绕图遍历、最短路径、最小生成树和拓扑排序展开。

### 2. 图存储结构

> 在图存储方面，我们实现了邻接矩阵和邻接表两种结构。邻接矩阵用二维数组保存边权，如果两个城市之间没有道路，就用无穷大表示。它的优点是查询任意两个城市之间是否有边非常快，时间复杂度接近 O(1)。邻接表则是每个城市维护一条边链表，只保存实际存在的道路，更适合城市路网这种稀疏图。

### 3. DFS 和 BFS

> 图遍历部分实现了 DFS 和 BFS。DFS 是深度优先，尽量沿着一条路径走到底，再回溯；BFS 是广度优先，用队列一层一层访问相邻城市。为了处理非连通图，我们不是只从一个起点遍历，而是在必要时继续扫描未访问城市，保证最终可以覆盖所有连通分量。

### 4. Dijkstra 和 Floyd

> 最短路径部分实现了 Dijkstra 和 Floyd。Dijkstra 用于单源最短路径，给定一个起点，计算它到其他城市的最短距离。算法维护 `dist` 数组和 `prev` 数组，每次选出当前未确定城市中距离最小的一个，然后用它去松弛邻接边。`prev` 数组用于最后回溯完整路径。

> Floyd 用于多源最短路径，它通过三重循环枚举中转点，判断经过某个中转点是否能让两个城市之间的距离更短。它可以一次性求出任意两座城市之间的最短距离。

### 5. Prim 和 Kruskal

> 最小生成树部分实现了 Prim 和 Kruskal。Prim 是从一个起点开始，不断选择连接已选顶点集合和未选顶点集合之间权值最小的边。Kruskal 是先把所有边按权值排序，然后从小到大尝试加入边，用并查集判断是否形成环。如果不会成环，就加入生成树。

> 这两个算法都只适用于无向图。因为最小生成树本身是无向连通图上的概念，有向图下不是这个问题，所以系统会在有向图执行 MST 时提示不适用。

### 6. 拓扑排序

> 拓扑排序使用的是 Kahn 算法。它只适用于有向图。核心思路是先统计每个顶点的入度，把入度为 0 的顶点入队；每次出队一个顶点，把它加入拓扑序列，并把它指向的邻接点入度减一。如果新的入度变成 0，就继续入队。最后如果输出数量小于顶点数，说明图中存在环。

### 7. 城市 ID 映射修复

> 项目后期我们修复了一个比较关键的问题：用户输入的城市 ID 不一定连续，比如可能是 10、20、30。以前部分算法直接把城市 ID 当作数组下标，这样会导致越界或结果错误。现在算法中会先通过 `get_all_vertex_ids()` 获取全部城市 ID，再建立 ID 到内部数组下标的映射。这样不管城市编号是否连续，算法都能正确运行。

### 8. 小结

> 总结一下，我负责的部分重点是把图论中的经典算法真正落到代码上，并适配项目中的两种图存储结构。通过这个项目，我对图的遍历、最短路径、最小生成树、拓扑排序以及并查集等数据结构有了更完整的实践理解。

## 三、自己负责部分的知识点

### 1. DFS 深度优先遍历

核心思想：

- 从起点开始访问
- 访问一个未访问邻接点
- 一直深入，直到没有新邻接点
- 再回溯

复杂度：

- 时间复杂度：O(V+E)
- 空间复杂度：O(V)

适用：

- 图遍历
- 连通性判断
- 路径搜索

答辩说法：

> DFS 的特点是“走到底再回退”，适合深度搜索路径。本项目中 DFS 会处理非连通图，避免只遍历到一个连通分量。

### 2. BFS 广度优先遍历

核心思想：

- 使用队列
- 先访问起点
- 再访问所有一层邻接点
- 再访问下一层

复杂度：

- 时间复杂度：O(V+E)
- 空间复杂度：O(V)

答辩说法：

> BFS 的特点是一层一层扩展，使用队列保证先发现的顶点先处理。

### 3. Dijkstra 单源最短路径

核心数据：

- `dist[i]`：起点到第 i 个城市的当前最短距离
- `prev[i]`：最短路径上第 i 个城市的前驱城市
- `visited[i]`：该城市最短距离是否已经确定

流程：

1. 初始化起点距离为 0，其他为无穷大
2. 每次选未确定城市中 `dist` 最小的城市
3. 遍历它的邻接边，尝试松弛
4. 重复直到所有点确定或剩余点不可达

复杂度：

- 当前数组版本：O(V² + E)，通常写作 O(V²)
- 如果用优先队列优化：O((V+E)logV)

限制：

- 不能处理负权边

答辩说法：

> 本系统道路权值代表距离或通行成本，输入时要求为正数，所以满足 Dijkstra 的使用条件。

### 4. Floyd 多源最短路径

核心思想：

> 枚举每个城市 k 作为中转点，判断 i 到 j 是否可以通过 k 得到更短路径。

状态转移：

```text
if dist[i][k] + dist[k][j] < dist[i][j]:
    dist[i][j] = dist[i][k] + dist[k][j]
```

复杂度：

- 时间复杂度：O(V³)
- 空间复杂度：O(V²)

答辩说法：

> Floyd 虽然复杂度高，但可以一次性求出任意两点之间的最短距离，适合城市数量不是特别大的场景。

### 5. Prim 最小生成树

核心思想：

- 从任意顶点开始
- 每次选择连接“已加入集合”和“未加入集合”的最小权边
- 加入新的顶点
- 重复直到所有顶点加入

复杂度：

- 当前数组实现：O(V²)

适用：

- 无向连通图

答辩说法：

> Prim 更像是从一个点开始不断扩展生成树，每一步都选择当前能连接出去的最小边。

### 6. Kruskal 最小生成树

核心思想：

- 收集所有边
- 按权值从小到大排序
- 依次尝试加入边
- 用并查集判断是否成环
- 不成环就加入 MST

复杂度：

- 排序理想情况下 O(ElogE)
- 并查集接近 O(1)
- 当前如果使用简单排序，排序部分还有优化空间

答辩说法：

> Kruskal 是按边来选，Prim 是按点集来扩展。两者都能求最小生成树，可以相互验证结果。

### 7. 并查集

用途：

- 判断两个顶点是否已经在同一个连通分量
- Kruskal 中用于判断加入边是否会形成环

核心操作：

- `find_root`
- `union_sets`
- `is_connected`

优化：

- 路径压缩
- 按秩合并

答辩说法：

> 并查集可以高效维护连通分量。Kruskal 中如果一条边的两个端点已经在同一个集合，再加入就会形成环。

### 8. 拓扑排序

适用：

- 有向无环图 DAG

核心数据：

- 入度数组
- 队列
- 输出序列

判断有环：

- 如果最后输出顶点数小于总顶点数，说明存在环

答辩说法：

> 拓扑排序可以理解为不断删除入度为 0 的点。如果最后还有点删不掉，说明这些点之间存在环。

### 9. 城市 ID 映射

为什么需要：

- 用户输入 ID 可能是 1、5、99
- 不能直接作为数组下标
- 数组需要连续下标

解决方法：

1. 获取全部城市 ID 数组
2. 算法内部使用数组下标
3. 需要输出时再映射回城市 ID

答辩说法：

> 城市 ID 是业务编号，数组下标是算法内部编号，不能混用。

## 五、老师可能问的问题与回答

### Q1：DFS 和 BFS 的区别是什么？

答：

> DFS 是深度优先，沿着一条路径走到底再回溯；BFS 是广度优先，用队列一层层访问。两者时间复杂度都是 O(V+E)，但访问顺序不同。

### Q2：Dijkstra 为什么不能有负权边？

答：

> Dijkstra 每次会确定当前距离最小的未访问点，并认为这个距离已经是最优。如果存在负权边，后面可能通过负权边让已确定的距离变小，从而破坏算法正确性。

### Q3：Floyd 能处理负权边吗？

答：

> Floyd 可以处理负权边，但不能有负权环。本项目道路权值表示距离或通行时间，所以系统限制权值为正数，不涉及负权边。

### Q4：Dijkstra 和 Floyd 怎么选择？

答：

> 如果只关心一个起点到其他点，用 Dijkstra；如果要知道任意两个城市之间的最短距离，用 Floyd。Floyd 更全面，但复杂度更高。

### Q5：Prim 和 Kruskal 有什么区别？

答：

> Prim 是从一个顶点开始扩展，每次选连接当前树和外部顶点的最小边；Kruskal 是从全局边集合出发，按边权从小到大选边，并用并查集防止成环。

### Q6：为什么 MST 不能用于有向图？

答：

> 最小生成树定义在无向连通图上。有向图对应的问题不是普通 MST，而是最小树形图等其他模型，所以系统中 Prim 和 Kruskal 只允许无向图执行。

### Q7：拓扑排序为什么要求有向图？

答：

> 拓扑排序表达的是任务或节点之间的先后依赖关系，这种依赖关系是有方向的。无向边没有前后方向，所以无向图不适用拓扑排序。

### Q8：怎么判断有向图中有环？

答：

> Kahn 算法中，如果不断取入度为 0 的点，最后输出的顶点数量小于总顶点数，说明剩下的点互相依赖，存在环。

### Q9：并查集在 Kruskal 中怎么用？

答：

> 每个连通分量是一个集合。处理一条边时，如果两个端点已经在同一个集合，加入这条边会成环；如果不在同一个集合，就可以加入，并合并两个集合。

### Q10：为什么要处理非连通图？

答：

> 真实路网可能不是完全连通的，测试数据也可能出现非连通情况。遍历算法需要覆盖所有连通分量，MST 和最短路径也要能正确提示不可达或不连通。

### Q11：无穷大 `INF_WEIGHT` 有什么作用？

答：

> 用来表示两个城市之间当前不可达或没有直接道路。最短路径初始化时，除起点外其他距离都设为无穷大。

### Q12：最短路径中怎么恢复路径？

答：

> Dijkstra 松弛时会记录前驱节点 `prev`。从终点开始不断沿 `prev` 回溯到起点，再反转顺序，就得到完整路径。

### Q13：城市 ID 非连续时为什么会出错？

答：

> 如果把城市 ID 当数组下标，比如 ID 是 100，就会访问数组第 100 位，但实际可能只有 3 个城市，导致越界或错误。所以要建立 ID 到内部下标的映射。

### Q14：你实现的算法是否依赖邻接矩阵？

答：

> 不依赖。算法通过 `GraphBase` 获取邻接点和顶点列表，所以可以运行在邻接矩阵和邻接表上。

### Q15：如果图很大，哪个部分需要优化？

答：

> Dijkstra 可以用优先队列优化到 O((V+E)logV)；Kruskal 的边排序可以用更高效的排序算法；Floyd 因为 O(V³)，大图下不适合频繁使用。

### Q16：BFS 为什么要用队列？

答：

> 队列是先进先出，可以保证先发现的顶点先处理，从而实现按层访问。

### Q17：DFS 为什么可能用栈？

答：

> DFS 本质上需要回溯。递归调用隐含使用系统调用栈，也可以显式使用顺序栈来实现非递归 DFS。

### Q18：图中自环边怎么处理？

答：

> 系统不允许自环边，因为道路从一个城市到自己没有实际意义，也会影响部分算法判断。输入时会返回 `ERR_SELF_LOOP`。

### Q19：如果两个城市之间没有路径，Dijkstra 怎么输出？

答：

> 距离保持为 `INF_WEIGHT`，输出时会提示不可达或显示 INF。

### Q20：你这部分最大的收获是什么？

答：

> 最大收获是把课堂上的图算法真正和项目业务结合起来，并且处理了很多实际问题，比如非连续城市 ID、非连通图、有向/无向图适用范围等。

## 六、如果被要求现场讲一个算法

### 最推荐讲 Dijkstra

短版：

> Dijkstra 从一个起点出发，维护起点到所有点的当前最短距离。每次选择未确定点中距离最小的点，将它标记为确定，然后用它的出边更新邻接点距离。因为所有边权为正，所以一旦某个点成为当前最小，它的最短距离就不会再被更小路径更新。

配合例子：

> 比如从城市 1 出发，先把 1 到相邻城市的距离更新。然后选择当前距离最小的城市继续扩展。最终得到 1 到所有城市的最短路径。

### 如果老师要求讲 Kruskal

短版：

> Kruskal 是按边权从小到大选边。每次选一条最小边，如果这条边连接的是两个不同连通分量，就加入生成树；如果两个端点已经连通，加入会成环，就跳过。并查集负责快速判断两个端点是否属于同一个集合。

### 如果老师要求讲拓扑排序

短版：

> 拓扑排序先统计所有点入度，把入度为 0 的点入队。每次取出一个入度为 0 的点，删除它的出边，相当于让邻接点入度减一。如果新的入度变为 0，就继续入队。最后输出所有点说明无环，否则有环。

## 七、老师专项六问：核心算法与业务功能

### 1. 本系统要求手写算法，请口述伪代码并解释算法执行流程。

如果老师没有指定算法，优先讲 Dijkstra；如果老师指定排序/查找/拓扑/MST，就切换到对应算法。

#### Dijkstra 伪代码

```text
Dijkstra(G, start):
    获取全部城市 ID，建立 ID 到下标的映射
    for 每个城市 i:
        dist[i] = INF
        prev[i] = INVALID
        visited[i] = false
    dist[start_index] = 0

    repeat V 次:
        u = 未访问城市中 dist 最小的城市
        if u 不存在: break
        visited[u] = true

        for u 的每条邻接边 edge(u, v, w):
            v_index = id_to_index(v)
            if dist[u] + w < dist[v_index]:
                dist[v_index] = dist[u] + w
                prev[v_index] = u 的城市 ID
```

解释：

> Dijkstra 每次确定一个当前距离最小的城市，然后用这个城市去更新它的邻接城市。因为本系统道路权值都是正数，所以当前最小距离一旦确定，就不会再被后续路径更新。`prev` 数组用于保存路径前驱，最后可以回溯最短路径。

#### Kruskal 伪代码

```text
Kruskal(G):
    收集图中所有边
    按边权从小到大排序
    初始化并查集
    for 每条边 e(u, v, w):
        if find(u) != find(v):
            加入这条边
            union(u, v)
        if 已选边数 == V - 1:
            break
```

解释：

> Kruskal 是按边权从小到大选择边。并查集用来判断两个端点是否已经连通，如果已连通，再加入这条边就会成环，所以要跳过。

#### 拓扑排序伪代码

```text
TopoSort(G):
    计算每个顶点入度
    入度为 0 的顶点入队
    while 队列非空:
        u = 出队
        输出 u
        for u 的每个邻接点 v:
            indegree[v]--
            if indegree[v] == 0:
                v 入队
    if 输出数量 < 顶点数量:
        图中有环
```

### 2. 分析核心算法时间复杂度、空间复杂度，是否存在优化空间？如何优化？

推荐回答：

| 算法 | 时间复杂度 | 空间复杂度 | 优化方向 |
| ---- | ---------- | ---------- | -------- |
| DFS | O(V+E) | O(V) | 基本已经最优 |
| BFS | O(V+E) | O(V) | 基本已经最优 |
| Dijkstra 数组版 | O(V²+E)，通常写 O(V²) | O(V) | 用最小堆优化为 O((V+E)logV) |
| Floyd | O(V³) | O(V²) | 大图下改为多次 Dijkstra |
| Prim 数组版 | O(V²) | O(V) | 用优先队列优化 |
| Kruskal | 取决于排序，理想 O(ElogE) | O(V+E) | 使用更高效排序替代简单排序 |
| 拓扑排序 | O(V+E) | O(V) | 基本已经最优 |

可以这样说：

> 优化空间最大的主要是 Dijkstra、Prim 和 Kruskal。Dijkstra 和 Prim 可以用最小堆优先队列优化，Kruskal 的关键在边排序，可以使用快排或归并排序替代简单排序。

### 3. 增删改查核心业务逻辑中，多条数据联动修改时如何保证数据一致性？举一处业务场景说明。

推荐回答：

> 路网编辑通过 `RoadNetwork` 统一管理，内部同时维护邻接矩阵和邻接表。比如添加道路时，要先检查起点、终点是否存在，权值是否合法，是否自环，是否重复。然后分别向矩阵和邻接表添加边。  
>
> 如果矩阵添加成功但邻接表添加失败，就要回滚矩阵中的边，避免一边有数据、一边没有数据。删除城市也是联动操作，因为删除一个城市时，所有和它相关的道路都要被清理。

可以举例：

> 删除城市 2 时，不只是删除顶点 2，还要删除其他城市指向 2 的道路，以及 2 指向其他城市的道路。邻接矩阵和邻接表都要同步更新。

### 4. 题目要求多种数据结构联用，你在哪块功能同时使用两种及以上结构？各自承担什么作用？

推荐回答：

> Kruskal 同时使用了边数组和并查集。边数组用于收集并排序所有道路，并查集用于判断加入某条边是否会形成环。  
>
> 拓扑排序同时使用图结构、入度数组和队列。图结构提供邻接关系，入度数组记录每个城市当前依赖数量，队列保存入度为 0 的城市。  
>
> Dijkstra 同时使用图结构、距离数组、前驱数组、访问标记数组和 ID 映射。图结构提供邻接边，数组保存算法状态，ID 映射解决城市编号不连续问题。

### 5. 算法调试阶段遇到过逻辑死循环、查找结果错误等 BUG 吗？举例说明定位与修复过程。

推荐回答：

> 遇到过比较典型的查找结果错误，就是城市 ID 被直接当作数组下标使用。比如城市 ID 如果是 10、20、30，数组实际长度只有 3，但代码访问 `dist[20]` 就会越界或得到错误结果。  
>
> 定位时我们发现，算法中部分数组下标和业务 ID 混用了。修复方法是统一通过 `get_all_vertex_ids()` 获取城市 ID 列表，再用 `find_city_index()` 或映射函数把城市 ID 转成内部连续下标。输出时再把内部下标还原为城市 ID。  
>
> 修复后，DFS/BFS、Dijkstra、Floyd、Prim/Kruskal、拓扑排序都不再依赖连续城市 ID。

如果老师问“有没有死循环”：

> 拓扑排序中如果有环，不能一直等待入度变 0。我们通过判断最终输出顶点数是否小于总顶点数来识别环，避免逻辑卡住。

### 6. 为什么不直接调用 C++ 内置排序、查找函数？手写算法相比库函数有哪些优势与不足？

推荐回答：

> 首先这是数据结构课程设计，要求我们理解并手写核心算法和数据结构，所以不能只调用库函数。手写的优势是能清楚掌握算法过程、复杂度和每一步状态变化，也方便按照项目需求定制，比如错误码、调试输出、城市 ID 映射。  
>
> 不足是手写算法在性能和鲁棒性上通常不如标准库，比如 `std::sort` 经过大量优化，边界情况也处理得更成熟。所以在真实工程中会优先考虑标准库；但在课程设计中，手写更能体现我们对数据结构和算法的掌握。
