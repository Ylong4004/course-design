## 二、完整个人发言稿

### 1. 开始部分

> 我是光雨晨，主要负责本项目的架构设计和底层公共接口。这个系统的核心对象是图，所以我首先考虑的是怎样把“城市交通路网”抽象成清晰、可复用的图模型。我们把城市抽象为顶点，把道路抽象为带权边，权值可以表示距离、耗时或者通行成本。

### 2. 分层架构

> 整个系统采用分层设计，从上到下分为 UI 菜单层、CLI 命令行层、服务层、算法层、图存储层和公共基础层。这样设计的好处是每一层职责比较清楚：UI 和 CLI 只负责交互；服务层负责路网业务逻辑；算法层只关心图接口；存储层负责邻接矩阵和邻接表的具体实现。

> 这个分层结构降低了耦合度。比如 Dijkstra 算法不需要知道当前图是邻接矩阵还是邻接表，它只需要调用 `GraphBase` 里的 `get_neighbors()`、`get_all_vertex_ids()` 这些接口。

### 3. GraphBase 抽象接口

> 我在 `graph_base.h` 中设计了 `GraphBase` 抽象基类。它定义了图存储必须提供的统一能力，比如添加顶点、删除顶点、添加边、删除边、查询边权、获取邻接点、获取全部城市 ID、统计性能数据等。邻接矩阵和邻接表都继承这个接口并分别实现。

> 这样做的核心意义是“算法和存储解耦”。如果以后要增加新的图存储结构，只要实现 `GraphBase` 接口，算法层几乎不需要修改。

### 4. 邻接矩阵与邻接表双存储

> 我们没有只选择一种图结构，而是同时维护邻接矩阵和邻接表。邻接矩阵适合快速查询任意两点之间是否有边，时间接近 O(1)，但空间复杂度是 O(V²)。邻接表只存实际存在的边，更适合城市路网这种相对稀疏的图。

> 为了保证两套数据一致，我设计了 `RoadNetwork` 作为统一业务入口。用户添加城市、删除城市、添加道路、删除道路、修改权值时，都通过 `RoadNetwork` 同步写入邻接矩阵和邻接表。如果其中一步失败，就要避免两边数据不一致。

### 5. 公共类型与错误码

> 在公共基础层，我们用 `City_t` 表示城市顶点，用 `Edge_t` 表示道路边，用 `GraphType` 表示有向图和无向图，用 `StorageType` 表示邻接矩阵和邻接表。错误处理采用统一错误码，例如城市不存在、城市重复、道路已存在、文件打开失败、格式错误等。这样菜单层和 CLI 层可以根据错误码输出更明确的中文提示。

### 6. 有向图/无向图加载优化

> 项目后期我们遇到一个实际问题：如果 JSON 文件中 `graph_type` 是 1，也就是有向图，但程序启动时默认创建的是无向图对象，那么加载时会因为类型不一致而失败。这个问题不是文件格式错，而是内存中的图对象类型和文件图类型不匹配。

> 为了解决这个问题，我们增加了 `FileManager::detect_graph_type()`，加载前先读取 JSON 里的 `graph_type`，然后调用 `RoadNetwork::reset()` 按有向或无向类型重建两套空图，再加载数据。与此同时，拥堵模拟器和结构对比器也要重新绑定新的图对象，避免继续指向旧图。

### 7. 小结

> 总结一下，我这部分工作的重点是把系统的骨架搭起来，让图存储、算法、业务逻辑和交互层之间通过清晰接口协作。这样项目后续增加 JSON 文件、新建路网、有向图加载等功能时，不需要大面积重写算法，而是在服务层和文件层扩展即可。

## 三、自己负责部分的知识点

### 1. 图的基本抽象

- 顶点：城市
- 边：道路
- 权值：距离、时间或成本
- 有向图：道路有方向，只能从 `from` 到 `to`
- 无向图：道路双向通行
- 带权图：每条边有权值

答辩时可以说：

> 城市交通路网天然适合用带权图表示。城市是顶点，道路是边，道路长度或通行时间是权值。

### 2. 邻接矩阵

知识点：

- 使用二维数组 `matrix[i][j]` 表示从顶点 i 到顶点 j 的边权
- 无边时一般用 `INF_WEIGHT` 表示
- 边查询快：O(1)
- 空间占用固定：O(V²)
- 适合稠密图

可能被问：

> 为什么邻接矩阵空间复杂度是 O(V²)？

回答：

> 因为它要为任意两个顶点之间是否有边都预留一个位置，所以 V 个顶点需要 V×V 的矩阵。

### 3. 邻接表

知识点：

- 每个顶点保存一条边链表
- 只保存实际存在的边
- 空间复杂度接近 O(V+E)
- 遍历邻接点方便
- 查询任意两个点是否相连需要遍历对应链表

可能被问：

> 城市交通路网更适合哪种结构？

回答：

> 一般更适合邻接表，因为城市之间不是任意两点都有道路，图比较稀疏。邻接表只存实际存在的边，空间更节省。

### 4. GraphBase 抽象类

知识点：

- C++ 抽象基类
- 纯虚函数
- 多态
- 面向接口编程
- 算法层依赖抽象而不是具体实现

可以说：

> `GraphBase` 相当于图存储层的统一协议。算法层只认识这个协议，不关心底层实现，所以可以同时适配邻接矩阵和邻接表。

### 5. RoadNetwork 双存储同步

知识点：

- 内部有两个 `GraphBase*`
- 一个指向邻接矩阵
- 一个指向邻接表
- 添加/删除/修改都同步执行
- 保持两种结构数据一致

重点说法：

> 用户不直接操作矩阵或链表，而是通过 `RoadNetwork` 统一入口操作，这样能保证两套结构始终一致。

### 6. `RoadNetwork::reset()`

知识点：

- 用于运行时切换有向/无向图
- 删除旧的矩阵图和邻接表图
- 按新 `GraphType` 重新创建
- 当前数据清空
- 加载不同图类型的 JSON 前会调用它

注意说明：

> `reset()` 不是把无向边转换成有向边，而是重建一个空的对应类型路网，再从文件加载数据。

### 7. 文件格式 JSON

知识点：

```json
{
  "graph_type": 0,
  "cities": [
    {"id": 1, "name": "City1"}
  ],
  "roads": [
    {"from": 1, "to": 2, "weight": 10}
  ]
}
```

- `graph_type=0`：无向图
- `graph_type=1`：有向图
- `cities`：城市数组
- `roads`：道路数组

### 8. 错误码设计

知识点：

- `SUCCESS`
- `ERR_CITY_NOT_FOUND`
- `ERR_CITY_DUPLICATE`
- `ERR_ROAD_EXISTS`
- `ERR_FILE_OPEN_FAIL`
- `ERR_FILE_FORMAT`
- `ERR_GRAPH_EMPTY`

可以说：

> 统一错误码可以让底层不直接输出界面信息，而由 UI 或 CLI 决定怎么提示用户。

## 五、老师可能问的问题与回答

### Q1：你们项目为什么要分层？

答：

> 分层可以降低耦合度。UI 只管输入输出，服务层处理业务逻辑，算法层只依赖图接口，存储层负责具体数据结构。这样某一层修改时，不会影响整个系统。

### Q2：为什么不用一个类把所有功能都写完？

答：

> 如果都写在一个类里，代码会很难维护，算法、文件、菜单、数据结构会混在一起。分模块以后，每个人可以负责不同部分，也方便测试和定位问题。

### Q3：GraphBase 里为什么要有那么多接口？

答：

> 因为算法和服务层需要的操作比较完整，比如添加顶点、删除顶点、获取邻接点、查询边权、统计性能等。把这些能力抽象出来，邻接矩阵和邻接表都按同一标准实现，算法层就可以统一调用。

### Q4：邻接矩阵和邻接表数据不一致怎么办？

答：

> 正常操作都通过 `RoadNetwork` 同步执行，不让上层直接分别修改两套结构。如果某一步失败，会返回错误码，必要时回滚已经执行的操作，尽量保持一致。

### Q5：为什么要用错误码，不用异常？

答：

> 这个项目是课程设计，错误情况比较明确，用错误码更直观，也方便菜单和 CLI 根据错误码输出中文提示。异常机制更适合复杂系统，但在这个项目中错误码足够清晰。

### Q6：你们怎么判断文件是有向图还是无向图？

答：

> JSON 文件里有 `graph_type` 字段。加载前先调用 `detect_graph_type()` 解析这个字段，0 表示无向图，1 表示有向图。

### Q7：为什么修改 `graph_type` 为 1 后以前会失败？

答：

> 因为程序原来默认创建无向图对象，文件里是有向图，加载时做类型校验就失败。现在已经改成加载前先识别文件类型，再重建对应图对象。

### Q8：`reset()` 会不会导致内存泄漏？

答：

> 不会。`reset()` 中会先通过 `safe_delete` 删除旧的矩阵图和邻接表图，再创建新的对象。析构时也会释放两套图结构。

### Q9：为什么重置路网后模拟器和对比器也要更新？

答：

> 因为它们内部保存的是图对象指针。`RoadNetwork::reset()` 会删除旧图并创建新图，如果不更新，模拟器和对比器就可能继续指向已经释放的旧对象。

### Q10：为什么要支持从 `bin/` 目录运行？

答：

> 用户经常会直接双击 `bin/network_console.exe`。这时工作目录是 `bin`，如果只找 `./data` 就会失败。所以文件管理模块会向上查找项目根目录中的 `data` 文件夹。

### Q11：CMake 在项目中起什么作用？

答：

> CMake 用来统一管理多文件编译。项目源文件比较多，手动写 g++ 命令容易漏文件。CMake 可以生成构建目标，比如纯控制台版 `traffic_console`，也可以可选构建 Qt GUI 版本。

### Q12：项目中哪个设计你觉得最重要？

答：

> 我认为最重要的是 `GraphBase` 抽象接口和 `RoadNetwork` 双存储管理。前者让算法和存储解耦，后者保证两套存储数据一致。

### Q13：如果以后加一个“邻接多重表”存储结构，要改哪里？

答：

> 只需要新增一个类继承 `GraphBase` 并实现所有接口。算法层基本不需要改，只要服务层选择新的存储对象即可。

### Q14：这个项目有什么不足？

答：

> JSON 解析器是针对本系统固定格式写的，不是完整通用 JSON 库；另外测试框架还没有完全接入 CTest，一键自动化测试还可以继续完善。

### Q15：如果老师问你具体贡献，怎么简短回答？

答：

> 我主要完成了系统分层架构、图抽象接口、公共类型和错误码、路网双存储同步管理、运行时图类型重置，以及部分文件加载流程和构建文档整理。

## 七、组员 A 专项提问准备：系统架构与数据结构

### 1. 结合选题说明整体业务流程，为什么选用这套数据结构组合？对比其他结构有什么优缺点？

推荐回答：

> 业务流程是：程序启动后加载 JSON 路网文件，将城市读成顶点，将道路读成带权边；用户可以通过菜单或 CLI 编辑城市和道路；之后可以执行遍历、最短路径、最小生成树、拓扑排序、拥堵模拟和性能对比；退出时保存当前路网。  
>
> 数据结构上，我们选择邻接矩阵 + 邻接表双结构。邻接矩阵查询任意两点之间是否有边很快，适合边查询和性能对比；邻接表只存实际存在的边，更适合城市路网这种稀疏图。  
>
> 其他可选结构比如单纯边集数组，适合 Kruskal 这类按边排序的算法，但不适合快速获取某个城市的邻接点；如果只用邻接矩阵，空间会浪费；如果只用邻接表，任意两点边查询不如矩阵直接。所以我们保留两套结构，并通过 `RoadNetwork` 保证同步。

### 2. 自定义全局结构体 / 结点各字段代表什么？设计时考虑了哪些数据约束？

推荐回答：

> `City_t` 表示城市顶点，包含 `id` 和 `name`。`id` 是城市唯一编号，`name` 是城市名称。  
>
> `Edge_t` 表示道路边，包含 `from`、`to`、`weight`，分别表示起点城市 ID、终点城市 ID 和道路权值。  
>
> `EdgeNode_t` 是邻接表中的边结点，保存目标城市、边权和下一条边指针。`AdjListHead_t` 是邻接表顶点头结点，保存城市信息和第一条边。  
>
> 设计约束包括：城市 ID 唯一，城市名不能为空且长度有限；道路不能自环，权值必须为正数且小于无穷大；有向图只保存单向边，无向图需要体现双向关系；文件中道路端点必须在城市列表中存在。

### 3. 公共文件读写接口实现逻辑是什么？如何保证多模块调用文件时不发生数据错乱？

推荐回答：

> 文件读写统一封装在 `FileManager` 中。保存时，从图对象中取出全部城市和道路，写成 JSON；加载时，先解析 JSON，校验 `graph_type`、城市数组和道路数组，再清空当前图并重新添加城市和道路。  
>
> 为避免数据错乱，上层不直接自己读写文件，而是统一调用 `FileManager::save_to_file()` 和 `load_from_file()`。对于不同图类型，先通过 `detect_graph_type()` 识别文件类型，再通过 `RoadNetwork::reset()` 创建匹配图结构。矩阵和邻接表分别加载同一份文件，保证两套结构一致。

### 4. 程序整体流程图、模块调用图如何体现模块关系？高内聚低耦合如何实现？

推荐回答：

> 程序入口在 `main.cpp`，进入 `cli_run()` 后根据参数选择菜单模式、CLI 模式或单条命令批处理。菜单和 CLI 都调用服务层，服务层再调用算法层和图存储层。  
>
> 高内聚体现在每个模块只做自己的事：算法模块只实现算法，文件模块只负责读写，菜单模块只负责交互。低耦合体现在模块之间通过接口通信，比如算法只依赖 `GraphBase`，UI 只依赖 `RoadNetwork` 和服务函数，不直接访问矩阵数组或链表结点。

### 5. 系统全局变量、基础公共函数为什么这样设计？如果新增业务功能，现有框架是否容易扩展？

推荐回答：

> 全局公共类型和错误码集中放在 `types.h`，安全内存宏和调试宏放在 `defines.h`，这样所有模块使用同一套定义，避免重复定义和含义不一致。  
>
> 扩展性比较好。比如新增一个“按城市名称查询”功能，可以在 `RoadNetwork` 中增加查询接口，再由菜单和 CLI 调用；新增一种图存储结构，只要继承 `GraphBase` 实现接口，算法层基本不用改。

### 6. 动态链表 / 树结点在哪里释放？如果不释放会出现什么问题？

推荐回答：

> 本项目中主要的动态链表结点是邻接表的 `EdgeNode_t`，它在删除边、删除顶点、清空图和邻接表析构时释放。最小生成树结果使用动态数组保存边集，使用后通过对应释放函数释放。算法中临时申请的邻接点数组、路径数组也在使用后 `delete[]`。  
>
> 如果不释放，会造成内存泄漏。程序运行时间长或频繁加载、切换路网时，内存占用会不断增加；如果释放后还继续访问旧指针，则会出现悬空指针，可能导致崩溃或错误结果。
